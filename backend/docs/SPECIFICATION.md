# システム仕様書 (System Specification)

## 1. システム概要 (Overview)

本システムは、機械学習モデルに対する **Membership Inference Attack (MIA)** などの実験を管理・自動実行するためのバックエンドAPIサーバーです。
実験のパラメータ（ハイパーパラメータやデータセットの分割比率など）を管理し、非同期タスクキューを用いて重い実験処理をワーカーに委譲します。また、実験結果のメトリクスや生成されたアーティファクト（MinIO上のパス）を一元管理します。

## 2. アーキテクチャ (Architecture)

本システムはRustで実装されており、以下の技術スタックとアーキテクチャを採用しています。

### 技術スタック

- **Web Framework**: Axum
- **ORM**: SeaORM (PostgreSQL / MySQL等に対応)
- **Task Queue**: Celery (Broker: Redis)
- **Object Storage**: MinIO (S3互換ストレージ、アーティファクト保存用)

### レイヤードアーキテクチャ

システムは以下のレイヤーに分割されています。

1. **Handlers (`src/handlers/`)**: HTTPリクエストの受け取り、バリデーション、レスポンスの返却。
2. **Services (`src/services/`)**: ビジネスロジックの実行、トランザクションや複数リポジトリのオーケストレーション。
3. **Repositories (`src/repositories/`)**: データベースやRedis（Celery）へのアクセス抽象化。
4. **Entities (`src/entities/`)**: データベースのテーブルと1対1で対応するモデル定義。
5. **DTO (`src/dto/`)**: APIリクエスト/レスポンス用のデータ転送オブジェクト。

## 3. データモデル (Data Models)

### Experiment (実験)

実験の条件、進行状態、および結果を保持する主要なエンティティです。

- **基本情報**: ID, 実験名 (`name`), 備考 (`notes`), 攻撃手法 (`method`: `offline_lira`, `shokri`)
- **実験条件**: バッチサイズ, 最大エポック数, シャドウモデル数, 各種データサイズ, シード値, その他ハイパーパラメータ(JSON)
- **データ流用**: ベースとなる実験ID, 各種モデルのロードフラグ
- **状態管理**: ステータス (`WAITING`, `RUNNING`, `SUCCEEDED`, `FAILED`), 作業PC名, 完了日時, エラーメッセージ
- **実験結果**: 全体AUC, 各FPRにおけるTPR (1%, 0.1%, 0.01%), 実行時間
- **ファイルパス**: MinIOベースパス, データセットJSONパス, 実行ログパス, その他ファイルパス

### Task (タスク)

Celeryキューに登録される非同期タスクの情報を表します。

- **基本情報**: タスクID (UUID), タスク名
- **引数情報**: 位置引数, キーワード引数, 制御情報
- **エラー情報**: エラーメッセージ

## 4. APIエンドポイント (API Endpoints)


| メソッド     | エンドポイント             | 説明                                    |
| -------- | ------------------- | ------------------------------------- |
| `GET`    | `/experiments`      | 登録されているすべての実験一覧を取得します。                |
| `POST`   | `/experiments`      | 新しい実験を作成し、Celeryに非同期タスクをエンキューします。     |
| `PUT`    | `/experiments`      | ワーカーが実験結果や生成ファイルのパスをシステムに反映（完了報告）します。 |
| `DELETE` | `/experiments/{id}` | 指定されたIDの実験を削除します。                     |
| `GET`    | `/tasks`            | 現在キューイングされている、または実行中のタスク一覧を取得します。     |
| `DELETE` | `/tasks/{id}`       | 指定されたIDのタスクを削除（キャンセル）します。             |


## 5. 非同期タスクフロー (Asynchronous Task Flow)

実験の実行は、APIサーバーとワーカー（別プロセス）の間で非同期に行われます。

1. **実験の登録 (API)**
  - ユーザーが `POST /experiments` に実験条件を送信します。
  - システムはDBに `Experiment` レコードを `WAITING` 状態で作成します。
  - 成功後、Celeryブローカー（Redis）に実験実行タスクをエンキューします。
2. **タスクの実行 (Worker)**
  - ワーカープロセスがRedisからタスクをフェッチし、機械学習の実験（MIAなど）を開始します。
  - ワーカーは必要に応じてMinIOから既存モデルをロードし、実験結果やログをMinIOにアップロードします。
3. **結果の反映 (Worker -> API)**
  - 実験が完了すると、ワーカーは `PUT /experiments` を呼び出し、評価メトリクス（AUCなど）とMinIOのファイルパスを送信します。
  - システムは該当する `Experiment` レコードのステータスを `SUCCEEDED` に更新し、結果を保存します。
