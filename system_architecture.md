# 分散タスク処理システム 設計書

## 1. システム全体像

本システムは、複数台のPCリソースを有効活用して機械学習やセキュリティ評価のプログラム（mia_attack 等）を効率的に並列実行するための「マスター・ワーカー型」分散処理基盤です。

Celery + Redisをベースとし、共有ストレージへのアクセス制限やファイル競合問題を解決するために、ファイル入出力をMinIO経由で行うアーキテクチャを採用しています。

## 2. ノード構成と役割

### 2.1 マスターノード (nn02: 133.44.75.85)
タスクの管理とデータの集約を担う中央サーバーです。
* **Redis (キュー)**: タスクの実行条件と、ワーカーのキュー待ち行列を管理します。
* **MinIO (オブジェクトストレージ)**: `kslib` (NAS) をバックエンドの保存先としてマウントし、他ノードに対してHTTPベースでファイルのアップロード/ダウンロードAPIを提供します。これにより、`kslib`へ直接アクセスできないPCからのデータ入出力を仲介します。
* **可視化・管理サーバー (将来構築)**: タスクの投入や、MinIO・DBの情報を可視化する自作ダッシュボード。

### 2.2 ワーカーノード (nn01, nn02, nn11, t43)
タスクを取得し、実際の演算処理を行うPC群です。（nn02はマスター兼ワーカーとなります）
* **Celeryワーカー (Dockerコンテナ)**:
  * 環境構築済みのDockerコンテナ内で起動しっぱなし（常駐）にします。
  * 本部(nn02)のRedisを常に監視し、空きがあればタスクを取得します。
  * 実行時は、タスクごとにローカルディスク（`/tmp/task_<UUID>`など）に一時ディレクトリを作成し、共有ストレージ(NAS)上でのファイル競合を防ぎます。

## 3. タスク実行のワークフロー

1. **タスク投入**: ユーザーまたは管理スクリプトが、実行条件（ハイパーパラメータ等）をnn02のRedisキューに登録する。
2. **タスク取得**: 待機中のワーカー（Celery）がRedisから条件を取得する。
3. **環境準備 (キャッシュ機能)**: 
   * ワーカーはローカルに一時実行ディレクトリを作成する。
   * 必要なデータセットやモデルがローカルキャッシュにない、またはハッシュが古い場合は、nn02のMinIOからダウンロードする。
4. **プログラム実行**: コンテナ内の同一Python環境下で、`main.py` に相当する処理を呼び出して実行する。
5. **結果保存**:
   * **成功時**: 実行結果（約25MB）をnn02のMinIOへアップロードし、ステータスを「完了」とする。
   * **エラー時**: 実行結果データは破棄し、スタックトレースやPC名などをエラーステータスとして記録する。
6. **後処理**: ローカルの一時実行ディレクトリを削除し、次のタスクを待機する。

## 4. 懸念事項の解決策（振り返り）
* **NASでのファイル衝突**: 実行時のファイル書き出しを各PCのローカルディスクに行うことで解決。
* **ネットワークアクセス制限**: `kslib` にアクセスできないPCも、nn02に立てたMinIO(HTTP)を経由することでアクセス可能に。
* **モデルのアーキテクチャ互換性 (arm64/amd64)**: Dockerfileを共通化し、PyTorchのモデル保存・読み込みを `state_dict` 形式に統一することで解決。
* **帯域・ストレージ圧迫**: 各ワーカーにデータキャッシュの仕組みを実装することで、無駄なダウンロードを防止。

## 5. 今後すべきこと（実装ロードマップ）

この設計に基づいて、以下のフェーズで実装を進めていきます。

### Phase 1: マスターノード (nn02) のインフラ構築
- [ ] nn02上で `docker-compose.yml` を作成し、**Redis** コンテナを立ち上げる。
- [ ] nn02上で **MinIO** コンテナを立ち上げ、データ保存先を `kslib` のマウントパスに指定する。

### Phase 2: PythonコードのCelery対応（ワーカー実装）
- [ ] 必要なライブラリ（`celery`, `redis`, `boto3` または `minio`等）を `Dockerfile` および `Dockerfile.arm64` に追加する。
- [ ] `tasks.py` (仮) を作成し、Celeryアプリケーションを定義する。
- [ ] 既存の `main.py` を呼び出せるように、Celeryの `@app.task` デコレータをつけたラッパー関数を作成する。

### Phase 3: データ連携とキャッシュ機能の実装
- [ ] タスク開始時に、MinIOから必要なデータをダウンロードする関数を実装する。（ハッシュ比較によるキャッシュロジックも組み込む）
- [ ] タスク終了時に、結果をMinIOへアップロードする関数を実装する。
- [ ] 実行時の一時ディレクトリ作成・削除（クリーンアップ）ロジックを実装する。

### Phase 4: テスト実行とワーカーのデプロイ
- [ ] 1台のPCでワーカーコンテナを起動し、簡単なタスクが成功・失敗（エラーハンドリング）するかテストする。
- [ ] 他のPC (nn01, nn11, t43) でもワーカーコンテナを立ち上げ、並列でタスクを消化できるか確認する。

### Phase 5: 可視化・管理システムの開発 (オプション)
- [ ] MinIOやRedis(または追加したDB)から情報を読み取り、タスクの進捗や結果を確認できるダッシュボードを自作する。
